# 安全策略

## 支持版本

仓库只对受支持的主干分支状态提供安全修复支持。

| 版本 | 是否支持 |
| --- | --- |
| `master` | 是 |
| 更早提交或分叉版本 | 否 |

## 漏洞报告方式

发现疑似安全漏洞时，不要直接提交公开 GitHub Issue。

请使用以下方式之一：

1. 仓库启用时，使用 GitHub 私有漏洞报告
2. 直接联系仓库维护者

报告中请至少包含：

- 受影响模块
- 精确版本或 commit
- 复现步骤
- 影响评估
- 是否影响机密性、完整性、可用性或运行隔离

## 响应预期

- 首次确认目标：5 个工作日内
- 分诊结果：确认、拒绝或要求补充细节
- 修复策略：优先在 `master` 上修复；若公开行为发生变化，同步更新 `README.md` 或相关运维文档

## 关注范围

本项目是撮合引擎与撮合节点服务，安全敏感区域包括：

- gRPC TLS 与 mTLS 配置
- ZooKeeper lease / fencing 处理
- 服务发现元数据
- WAL、快照与重放完整性
- readiness、failover 与复制控制路径

## 部署边界

项目默认假设撮合节点部署在受控内网中，例如专用 VPC、私有子网、安全组、内网网关或同机 sidecar 后方。HTTP 默认绑定 `127.0.0.1`，不建议直接暴露到公网。

当 HTTP 绑定非 loopback 地址时，生产模式必须显式配置 `matcher.allowInsecureRemoteHttp=true`。该开关表示部署方已经确认 HTTP 明文入口只在受控内网中使用。跨公网、跨不可信网络或多租户边界时，应在外层网关启用 TLS、认证、限流和访问控制。
