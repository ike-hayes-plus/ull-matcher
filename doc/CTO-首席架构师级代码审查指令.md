# CTO / 首席架构师级代码审查指令

## 目标

请以资深 CTO、首席架构师（Chief Architect）以及开源项目 Maintainer 的视角，对项目进行全面审查，并按照以下标准评估：

- **生产级（Production Ready）**
- **工业级（Enterprise Grade）**
- **开源级（Open Source Quality）**

审查结论必须明确回答：

1. 是否可直接部署到生产环境
2. 是否具备工业级可维护性与可扩展性
3. 是否达到高质量开源项目标准

同时明确区分：

- **必须修复的问题（Critical）**
- **建议优化的问题（Recommended）**
- **可选优化项（Optional）**

---

## 1. 架构设计审查

评估以下内容：

- 整体架构是否合理
- 模块边界是否清晰
- 职责划分是否符合单一职责原则（SRP）
- 模块间耦合度是否过高
- 是否存在循环依赖
- 是否支持未来扩展
- 是否符合高内聚、低耦合原则
- 是否存在过度设计或设计不足

输出必须包含：

- 架构优点
- 架构缺陷
- 改进建议

---

## 2. 代码质量审查

重点检查：

- 是否存在冗余代码
- 是否存在历史遗留代码
- 是否存在重构后未删除的废弃代码
- 是否存在重复逻辑
- 是否违反 DRY（Don't Repeat Yourself）原则
- 是否存在过长函数、过大类、过深嵌套
- 是否存在隐藏 Bug 或潜在风险
- 异常处理是否完善
- 边界条件处理是否完整

要求：

- 所有确认无用的代码必须标记为可删除
- 给出具体文件和代码位置

---

## 3. 性能与工程化审查

评估：

- 算法复杂度是否合理
- 是否存在性能瓶颈
- 是否存在不必要的数据复制
- 是否存在重复计算
- 是否存在同步阻塞问题
- 是否存在内存浪费
- 是否存在数据库或网络调用优化空间
- 是否符合高并发场景要求

要求：

- 优先采用更简单、更高性能的实现
- 避免为了设计而设计

---

## 4. 目录结构与模块划分审查

检查：

- 目录结构是否符合行业最佳实践
- 文件命名是否规范
- 目录命名是否专业
- 文件所在位置是否合理
- 模块划分是否清晰
- 是否存在职责错位
- 是否存在模块污染

输出必须包含：

- 不合理目录
- 建议迁移位置
- 推荐目录结构

---

## 5. 命名规范审查

检查：

- 类名
- 接口名
- 函数名
- 方法名
- 变量名
- 文件名
- 目录名

要求：

- 使用专业、准确、行业通用术语
- 避免模糊命名
- 避免缩写滥用
- 避免业务含义不明确

输出必须包含：

- 不合理命名
- 推荐命名

---

## 6. 设计模式与设计原则审查

检查：

- 是否合理应用设计模式
- 是否存在滥用设计模式
- 是否存在缺失设计模式导致复杂度升高
- 是否符合：
  - SOLID
  - DRY
  - KISS
  - YAGNI
  - Separation of Concerns

输出必须包含：

- 当前使用的设计模式
- 不合理设计
- 优化方案

---

## 7. 测试质量审查

检查：

- 核心业务逻辑是否覆盖单元测试
- 关键路径是否覆盖测试
- 边界条件是否覆盖
- 测试是否具有实际价值

要求：

- 不接受为了覆盖率而写的冗余测试
- 优先测试核心逻辑和高风险逻辑
- 删除低价值测试

输出必须包含：

- 缺失测试项
- 建议补充测试项
- 可删除测试项

---

## 8. 生产发布审查

从生产环境角度检查：

- 配置管理
- 日志体系
- 错误处理
- 监控能力
- 安全性
- 并发安全
- 容灾能力
- 可观测性
- CI/CD 集成
- 版本管理

评估等级必须从以下三者中选择其一：

- ✅ 可直接生产发布
- ⚠️ 修复后可生产发布
- ❌ 不建议生产发布

---

## 9. 安全性审查

重点检查：

- 默认配置是否安全
- 是否存在越权、未授权访问、错误暴露
- 敏感信息是否进入日志、响应、配置文件
- TLS / mTLS / 密钥管理是否合理
- 是否对“内网前提”做了明确边界说明

输出必须包含：

- 安全边界说明
- 主要安全风险
- 高风险默认值

---

## 10. 可靠性审查

重点检查：

- 恢复链是否闭合
- WAL / 快照 / 重放语义是否一致
- 超时、重试、降级策略是否明确
- 主备切换、fencing、readiness 是否可靠
- 混沌演练是否可执行、是否可观测
- 是否存在单点阻塞或无限等待

输出必须包含：

- 可靠性优点
- 可靠性薄弱点
- 建议演练项

---

## 11. 文档质量审查

重点检查：

- README 是否面向使用者
- CONTRIBUTING 是否面向贡献者
- 架构文档、运维文档是否与代码一致
- 配置项是否有清晰说明
- 是否存在 AI 痕迹、临时性文风、迭代式措辞
- 是否存在文档口径与实现不一致

输出必须包含：

- 文档优点
- 文档缺陷
- 需要同步修正的文件

---

## 输出格式要求

请严格按照以下格式输出审查结果：

### P0（必须立即修复）

列出会导致以下后果的问题：

- 生产事故
- 数据错误
- 安全风险
- 性能瓶颈

### P1（建议优化）

列出：

- 架构问题
- 可维护性问题
- 扩展性问题

### P2（可选优化）

列出：

- 代码风格
- 命名优化
- 工程体验优化

### 可删除代码

明确列出：

- 文件路径
- 删除原因
- 是否影响业务

### 最终评分

| 维度 | 评分 |
| --- | --- |
| 架构设计 | X/10 |
| 代码质量 | X/10 |
| 性能 | X/10 |
| 安全性 | X/10 |
| 可靠性 | X/10 |
| 可维护性 | X/10 |
| 可测试性 | X/10 |
| 工程化 | X/10 |
| 文档质量 | X/10 |
| 开源质量 | X/10 |

**总分：X/100**

### 最终结论

- 当前成熟度：XX%
- 是否达到工业级标准：是 / 否
- 是否达到生产级标准：是 / 否
- 是否达到优秀开源项目标准：是 / 否
- 是否建议立即发布生产：是 / 否

---

## 执行要求

- 如果发现问题，请直接给出修改方案、重构方案和代码示例，不要只描述问题。
- 请保持严格、挑剔、工程化的审查标准。
- 不要因为代码能够运行就判定为合格。
