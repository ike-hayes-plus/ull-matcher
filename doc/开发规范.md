# 开发规范

## 目标

本规范用于约束 `ull-matcher` 的代码、模块、文档、测试与发布行为，保证项目持续向生产级、工业级、开源级标准演进。

面向对象：

- 核心开发者
- 代码审查者
- 外部贡献者

---

## 1. 总体原则

- 正确性优先于局部性能
- 恢复语义优先于临时便利
- 热路径优先于框架偏好
- 明确边界优先于隐式行为
- 少而硬的测试优先于大量低价值测试

禁止事项：

- 不允许绕过 `WAL -> sequence -> apply -> replay` 主链直接改订单状态
- 不允许把业务策略直接塞进撮合热路径
- 不允许为了“抽象漂亮”引入没有收益的层次
- 不允许保留确认无用的重构残留、废弃测试、历史代码

---

## 2. 模块边界规范

### `matcher-core`

职责：

- 撮合规则
- 订单簿
- 命令模型
- 事件模型

要求：

- 不依赖 HA、HTTP、gRPC、Spring、ZooKeeper、Nacos
- 保持单线程状态拥有模型
- 不承载业务治理逻辑

### `matcher-storage`

职责：

- WAL
- 快照
- 重放

要求：

- 新增持久化字段时必须定义恢复语义
- 不能写业务决策逻辑

### `matcher-runtime`

职责：

- MatchLoop
- 提交网关
- WAL durability mode

要求：

- 热路径保持短
- 语义变化必须配置化

### `matcher-ha*`

职责：

- 租约
- fencing
- failover
- replication
- readiness

要求：

- 不能破坏单线程 apply 模型
- 复制必须通过标准命令链进入状态机

### `matcher-server`

职责：

- 节点服务装配
- HTTP 业务入口
- gRPC 控制面装配
- 恢复、治理、观测

要求：

- 是撮合节点服务，不是统一网关
- 面向内网调用，但仍需要过载保护、错误分类、观测能力

### `matcher-spring-boot-starter`

职责：

- Spring Boot / Spring Cloud 装配

要求：

- 只做适配，不改写主链逻辑
- 不能把 Spring 依赖带回核心模块

---

## 3. 代码规范

### 命名

- 使用行业通用术语
- 名称必须体现职责
- 避免无上下文缩写

### 复杂度

- 单类过大时优先按职责拆分
- 方法明显过长时必须评估是否拆分
- 三层以上嵌套视为坏味道

### 常量与硬编码

- 明显复用值提取为常量
- 运行参数进入配置，不写死在主路径
- 路由、指标名、协议字段集中管理

### 注释

- 注释解释约束、原因、边界
- 禁止临时性、对话式、迭代式注释
- 注释必须面向仓库使用者和维护者

---

## 4. 测试规范

- 优先覆盖核心逻辑和高风险路径
- 不以覆盖率数字为目标
- 每个测试都要有明确系统价值

必测范围：

- 撮合核心规则
- WAL / 快照 / 重放
- 主备复制
- failover / fencing
- HTTP / gRPC 错误边界
- TTL 恢复语义

不建议：

- 大量重复的无价值框架测试
- 时序脆弱、收益很低的集成测试

---

## 5. 性能规范

热路径包括：

- `submit -> WAL -> ring -> MatchLoop`
- `replicate -> WAL -> ring -> apply`

要求：

- 避免无界等待
- 避免高频日志
- 避免不必要分配
- 避免将平台治理逻辑放进热路径

对于会改变语义的性能优化，必须做成可配置模式，并同步 README 说明默认值、语义和风险。

---

## 6. 文档规范

### `README.md`

- 面向使用者、部署者、接入方
- 必须与代码和配置一致
- 不写内部讨论过程

### `CONTRIBUTING.md`

- 面向贡献者
- 构建和测试口径必须与代码一致

### `doc/architecture/*`

- 面向维护者和架构评审
- 解释边界、约束、扩展点

### `doc/operations/*`

- 面向运维和演练执行者
- 必须给出执行步骤、入口、指标、失败信号

---

## 7. 日志与可观测性规范

- 使用统一日志门面
- 热路径不打印高频 INFO
- 关键日志建议带：
  - `nodeId`
  - `role`
  - `sequence`
  - `epoch`
  - `category`

指标变更时必须同步 README。

---

## 8. 提交与发布规范

提交前至少执行：

```bash
mvn test -q
mvn -Pstyle-check validate -q
```

涉及混沌用例时额外执行：

```bash
mvn -Pchaos-tests test -q
```

在项目达到“可发生产工业级专业版”之前，远端推送按当前仓库约定：

- 只保留一个提交
- 提交信息统一为：`项目初始化`

---

## 9. 审查要求

关键变更后至少确认：

1. 主备写流量边界是否仍然成立
2. WAL / snapshot / replay 语义是否仍闭合
3. `ACCEPTED` 语义是否被改变
4. 文档是否同步
5. 测试是否仍然稳定、聚焦、可维护
